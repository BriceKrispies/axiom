# Axiom demo gallery (static showcase)

A mobile-first showcase of Axiom's browser demos, published to GitHub Pages by
`.github/workflows/deploy-pages.yml` on every push to `main`.

The gallery is **not a crate**. Every browser demo is its own standalone
composition-leaf app under `apps/` (e.g. `apps/axiom-gravix`,
`apps/axiom-growth`, `apps/axiom-quintet`) with its own wasm bundle, page, and
tests. This directory keeps only the static landing grid plus the committed
single-file pages of the pure-TS games:

```text
apps/axiom-gallery/
  web/                     # the static showcase (copied verbatim into dist/)
    index.html             # mobile-first landing grid of demo cards
    gallery.js             # the demo manifest (DEMOS): id/title/blurb + page link
    styles.css             # shared dark, mobile-first styling
    <ts-game>/index.html   # committed single-file pages of the pure-TS games
                           # (soccer-penalty-kick, signal-runner, swipe-basketball,
                           #  heat-check, home-run, minimal-3v3, three-point) —
                           #  regenerate with `make gallery-<name>`
    home-run-css/          # standalone CSS-3D curiosity page (unlinked)
```

Every card in `gallery.js` is a plain link to `<id>/index.html`. There is no
shared shell, no shared wasm bundle, and no boot logic here anymore.

## Build & preview locally

`make gallery` packages the whole showcase (`scripts/package_gallery.py`): it
copies this `web/` into `dist/`, then for each demo app copies
`apps/axiom-<id>/web/` into `dist/<id>/` and builds that app's own
capability-detecting loader (`dist/<id>/axiom-loader.js` + wasm, with a
Binaryen wasm2js fallback in full mode) beside its page. Then it serves
`dist/`:

```sh
make gallery           # package every app bundle + serve at http://localhost:8000
make gallery-fast      # quick wasm-only bundles (no wasm2js fallback) — for iteration
make gallery-serve     # re-serve the already-built dist/ WITHOUT rebuilding
make gallery-build     # package into dist/ only, no serve
```

To iterate on ONE demo with hot reload, skip packaging entirely:

```sh
cargo run -p axiom-serve -- gravix     # build + serve + auto-reload (see CLAUDE.md)
```

## Adding a demo

1. Create the demo as its own app crate `apps/axiom-<id>/` (see any extracted
   demo for the pattern: `Cargo.toml` cdylib+rlib, `app.toml`,
   `tests/architecture.rs`, `web/index.html` importing `./pkg/axiom_<snake>.js`,
   an `overlay_start` mount, and a root `Cargo.toml` workspace member entry).
2. Add the pair `("<id>", "apps/axiom-<id>")` to `DEMO_APPS` in
   `scripts/package_gallery.py` (and the workspace-console list in
   `scripts/package_workspace.py` if it should appear there).
3. Add a card to `DEMOS` in `web/gallery.js` with `page: "<id>/index.html"`.

Pure-TS games instead commit their single-file page under `web/<id>/` here
(regenerated by their `make gallery-<name>` target) and add the card the same
way.

## Netplay

The netplay demo (`apps/axiom-netplay`) needs a live authoritative server
(`make netplay-dotnet`, or the Rust `make netplay-server`), which static Pages
cannot host. The deployed page reads `?relay=`/`?server=` so you can point it
at a server you run. With no server set the demo still boots and the keypad
works; it just can't connect to a peer.

## One-time setup

GitHub Pages must be enabled once in the repo: **Settings → Pages → Build and
deployment → Source: GitHub Actions**. After that, every push to `main` deploys.
