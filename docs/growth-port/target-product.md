# Target product — what we are building on Axiom

Condensed from a full requirements audit of the *Growth* repo (Godot 4.6 + C# + a C++ `sim_core` GDExtension). This is the destination; Axiom is the proposed new foundation. Nothing here exists in Axiom yet unless [`axiom-baseline.md`](axiom-baseline.md) says so.

## Product shape

Two tiers, owned by a deterministic simulator, with presentation/input as a thin shell:

- **Overworld** — a generated **macro planet**: a low-resolution whole-sphere region atlas (tectonic plates, elevation, moisture, derived temperature/biome). Built **once per session**, read-only baseline for play. Stored authoritatively in the sim as a compact `PlanetSurfaceAtlas`.
- **Game world** — a **micro, metre-scale, player-editable** local simulation streamed as chunks (16×16 cells, 1 m cells ≈ 256 m/chunk) around the player, generated by a *second* procedural pass seeded from the overworld atlas + world seed. The sim owns authoritative cell state; the renderer shows diffs.

The long-horizon game (deliberately downstream of the substrate): walk → dig/terraform → survive (needs/threats) → guardrailed emergence → spirit/possession with multigenerational time → emergent ecology → stylised art pass.

## Subsystems and "done" criteria

These are the things that must eventually exist. Each is a candidate Axiom layer/module/app — placement is decided in [`gap-analysis.md`](gap-analysis.md).

### Determinism & data model
- Same seed + pipeline → same world. A "max determinism" mode is **bit-identical** for fixed inputs.
- `WorldSeed` (value + noise params) and a `PlanetGenome` (orbital/physical knobs + material tags + schema version, derived values on demand) persist for the session.
- World state is serializable/versioned (save/load is later-phase but the model must support it).

### Overworld generation pipeline (data-driven stage list)
Stage order is **data**, not hardcoded. Canonical Growth `default_globe` order:
`topology(icosphere) → half_edge_mesh → region_neighbours → tectonic_plates → plate_properties → elevation → erosion(stream-power) → fit_land_coverage → moisture → triangle_values → priority_flood → river_downflow → river_flow → river_carve → terrain_mesh`.
Plus later climate stages: prevailing **wind field → moisture advection → rain shadow**.
- "Done" = a deterministic atlas with per-region plate/elevation/moisture/positions + CSR neighbours, queryable by `locate_region(unit_dir)` / `sample_surface`.

### Planet/preset model
- Presets (earthlike / ocean / dry) as **distributions + constraints + material weights**; a genome is *sampled* from a preset under physical constraints (insolation, surface temp, gravity, pressure ranges).

### Surface atlas + query API
- `PlanetSurfaceAtlas` owned for the session; fast `locate_region` (spatial index, not O(R) scan); `sample_surface` returns a small dict (region id + scalars + derived temperature/biome).

### Climate / hydrology / terrain
- Latitude+elevation temperature; ocean-distance + wind-advected moisture; rain shadow.
- Stream-power erosion on the region graph; priority-flood drainage; rivers (flow accumulation + valley carve); land-fraction fitting at sea level 0; edge-preserving elevation smoothing.
- A height-field-on-fixed-icosphere terrain model (scalars mutate; topology fixed for a generation).

### Game-world streaming
- `GameWorldLocalMap` (chunk ↔ unit_dir frame); atlas-seeded chunk generator running a **data-driven** per-chunk pipeline (`sample_macro → base_height → detail_noise → build_height_grid`); continuous (IDW) macro sampling, coherent detail noise, shared chunk-seam heights; focus-radius load + sim-side unload; edits persist across regen; chunk-edit save/load.

### Gameplay layers (all downstream)
- Player avatar + play camera + interaction ray; **dig/terraform intents** that mutate chunk cells and emit cell-changed diffs; a sim-owned inventory; survival needs/threats + place/build; guardrailed emergence (bias defs steer gen/spawn); spirit influence + possession + a sim-time-gate; ecology (template species → regional populations → local agent spawn, "LOD for life"); presentation (cel materials, lighting, biome tints, foliage LOD).

### Moddability
- Content + assembly are **data** (defs, profiles, pipeline stage lists); only runtime systems are compiled. Pipelines, presets, biomes, interactions, scene/profile bindings are all overridable by packs.

## What the target explicitly is *not* (scope guards)
- Not photoreal — stylised "cel-adjacent."
- Not, today, a defined minute-to-minute **gameplay loop**: Growth's own docs schedule the first real player verb (dig) and tension (survival) *after* the substrate. The substrate is the near-term deliverable; the loop is intentionally deferred. (This matters for Axiom: see the vertical-slice recommendation in [`roadmap.md`](roadmap.md).)

## The one structural mismatch to keep in mind
Growth's simulator is ordinary multi-threaded C++ with branches and a worker-thread worldgen job. Axiom's engine spine is **branchless, 100%-covered, single-threaded-deterministic, and bans `thread::spawn`**. Porting *intent* is easy; porting *code shape* is not. The gap analysis treats this as the central constraint, not a footnote.
